# Chrome 139 性能更新摘要

## 1. 执行摘要

Chrome 139 在性能领域带来了显著提升，重点关注沉浸式网页体验和 Android 资源管理。值得注意的是，WebXR 深度感知现已提供可自定义机制，以优化深度缓冲区处理，直接提升 AR/VR 应用的性能。此外，Android 上的后台页面冻结触发时间已由五分钟缩短为一分钟，从而对后台标签页和 worker 实现了更积极的资源节省。

## 2. 主要影响

### 技术影响

- **现有实现**：WebXR 应用现在可以更精细地调整深度感知以提升性能，可能需要更新以利用新的缓冲区选项。Android 网页应用的后台任务可能会更早被挂起，影响长时间运行的后台操作。
- **新能力**：开发者获得了对 WebXR 深度缓冲区行为的细粒度控制，可实现定制化的性能优化。Android 用户则受益于设备资源利用率的提升。
- **技术债务**：依赖于 Android 上长时间后台执行的应用可能需要重构，以适应更早的冻结时机。WebXR 应用应检查深度缓冲区的使用，以充分利用新的性能选项。

## 3. 风险评估

**关键风险**：
- 这些功能未发现破坏性更改或直接的安全问题。

**中等风险**：
- **弃用**：暂无公告，但更早的后台冻结可能间接弃用依赖较长后台活动的模式。
- **性能影响**：更激进的后台冻结若处理不当，可能中断后台任务。WebXR 深度缓冲区的更改可能需要测试以避免性能回退。

## 4. 推荐措施

### 立即行动

- 检查并更新 WebXR 应用，利用新的深度缓冲区自定义选项以获得最佳性能。
- 审查 Android 网页应用的后台活动依赖，确保关键任务能在一分钟内完成，或迁移到前台执行。

### 短期规划

- 监控用户反馈及 WebXR 和 Android 后台冻结更改的性能指标。
- 更新文档和入门材料，反映 Android 上新的后台冻结行为。

### 长期策略

- 投资于 Android 的自适应后台任务管理策略，以应对未来后台执行时间的进一步缩减。
- 持续关注 WebXR 性能特性，并使应用架构与不断演进的深度感知能力保持一致。

## 5. 功能分析

### WebXR depth sensing performance improvements（WebXR 深度感知性能改进）

**影响级别**：🟡 重要

**变更内容**：
在 WebXR 会话中，开放了多种新机制以自定义深度感知功能的行为，旨在提升深度缓冲区的生成或使用性能。主要机制包括可请求原始或平滑深度缓冲区及其他配置选项。

**意义说明**：
此更改使开发者能够根据实际场景选择最合适的深度缓冲区类型，从而减少不必要的处理，提高帧率或延长电池续航。

**实施建议**：
- 评估当前 WebXR 深度感知的使用情况，判断原始或平滑缓冲区哪种更适合您的应用。
- 更新会话初始化代码以请求所需的缓冲区类型。
- 在支持的设备上测试性能和视觉质量的影响。

### Faster background freezing on Android（Android 后台冻结加速）

**影响级别**：🟡 重要

**变更内容**：
将 Android 上后台页面（及相关 worker）的冻结时间从五分钟缩短至一分钟。

**意义说明**：
此更改大幅降低了后台资源占用，提升了设备电池续航和系统性能。但可能会影响依赖长时间后台处理的应用。

**实施建议**：
- 检查 Android 网页应用中是否存在需要超过一分钟才能完成的后台任务。
- 重构关键后台操作，使其能在新的时间窗口内完成，或在前台触发。
- 监控是否出现意外挂起或用户体验回退。

**参考资料**：
- [Tracking bug #435623337](https://issues.chromium.org/issues/435623337)
- [ChromeStatus.com entry](https://chromestatus.com/feature/5386725031149568)
